2. Insight & Novelty

2.1 Core Insight

🎯 核心洞察
问：为什么传统多机器人协调流水线如此"脆弱"？
答：传统流程——自然语言问题描述 → 专家将其转化为数学优化问题 → 设计算法 → 编写代码 → 部署——有三个致命缺陷：(1) 劳动密集——每一步都需要领域专家手动完成；(2) 非专家不可访问——门槛极高；(3) 缺乏灵活性——任务稍有变化（如增加触发条件）必须从头重做。

问：本文的核心洞察是什么？
答：大语言模型可以同时扮演"领域专家"和"代码工程师"的双重角色。关键在于不让 LLM 从零生成所有代码（容易产生幻觉），而是提供标准化的 API 函数库（元行为库）——LLM 只需像搭乐高积木一样组合调用这些函数。配合行为树作为中间表示（自然语言 → 行为树 JSON → Python 代码），使得整个流水线既灵活又可追溯。额外发现：模板标准化——仅通过将原始自然语言对齐到专家模板——就能单独提升约 15% 的任务成功率。

2.3 Novelty Breakdown

Novelty 1: 从自然语言到行为树的任务分析模块（Mission Analysis）

【解决的问题】传统方法需要专家手动将NL任务分解为子任务并确定执行顺序——这是高度专业化、重复性且易出错的过程。任务变化时必须重做。
【灵感来源】LLM 在文本理解和结构化输出方面的能力——结合行为树在机器人控制中的成熟应用（四状态+三种节点类型）
【具体设计】三阶段流水线——(1) 模板标准化：用专家模板将原始 NL 转换为结构化格式（Phase/Task 层次 + 字段补全）；(2) 依赖分析：LLM 提取所有原子任务并分析其拓扑关系（A 必须早于 B，或 A 和 B 可并行，或 C 在触发后才激活）；(3) 综合任务分析：将结果组织为完整的行为树（每个节点含 type/status/constraints/trigger_condition/finish_condition/robot_ids 等字段）。最后选择所有前置依赖已满足的"就绪"Action 节点送入代码生成。

Novelty 2: 基于元行为库的安全代码生成（Code Generation with Meta Behavior Library）

【解决的问题】LLM 直接生成机器人控制代码易产生幻觉（调用不存在的函数）、语法错误、物理不可行的操作。需要既保持 LLM 编排灵活性又确保代码正确性的机制。
【灵感来源】"Code as Policies"（Liang et al. 2023）的思想——将机器人能力封装为 API 函数，LLM 只负责编排调用而非从零生成
【具体设计】构建三个子库——(a) Goal Generation：visit_points/follow_targets/herd 三种基本行为的原语；(b) Goal Allocation：default 按 ID 分配 + minimal_conflict 最小化轨迹交叉策略；(c) Motion Generation：从起点到目标的避障路径。LLM 为每个"就绪"Action节点生成四段代码：机器人执行代码、触发条件代码、Action节点完成条件代码、整体任务完成条件代码——全部基于元行为库API组合调用。

Novelty 3: LLM-in-the-loop 的动态重规划

【解决的问题】真实多机器人执行中，触发条件满足或任务完成后需要重新规划——传统方法依赖人工干预，现有 LLM 方法多假设"一次性规划"。
【灵感来源】闭环控制——感知→决策→执行→反馈的循环，将 LLM 置于反馈回路中
【具体设计】执行循环中持续检查两种条件：(a) 触发条件满足→LLM更新依赖分析（移除当前任务+添加新任务）→重新生成行为树→选择新就绪节点→生成新代码；(b) 完成条件满足→LLM更新行为树状态（标记Success+级联更新父节点）→选择下一就绪节点→继续执行。这种"执行-监控-重规划"闭环使 LAN2CB 能处理动态变化的多步骤长时序任务——这是与传统 LLM 规划器（SayCan 等）的根本区别。

4. Task — What Problem Does It Solve?

4.1 Problem Formulation

核心问题 $$ \text{Given: } \mathcal{D}_{\text{NL}} \text{ (自然语言任务描述), } \mathcal{R} = \{r_1,\dots,r_N\} \text{ (N台机器人)} $$ $$ \text{Find: } \Pi = \{\pi_1,\dots,\pi_N\} \text{ such that } \forall r_i \in \mathcal{R},\ \pi_i \text{ 协同实现 } \mathcal{D}_{\text{NL}} \text{ 且无需人工干预} $$
$\mathcal{D}_{\text{NL}}$ = 用户提供的自然语言任务描述（可能包含编队、区域访问、目标驱赶、触发条件、约束等） |  $\mathcal{R}$ = 多机器人团队（N 台机器人，实验中 5~10 台） |  $\Pi$ = 每台机器人的控制策略（Python 代码） |  核心约束：(1) 零人工干预——从 NL 到执行代码全自动；(2) 泛化性——覆盖 coordination strategies / motion patterns / constraints / triggers / finish conditions / mission finish conditions 六维任务空间；(3) 灵活性——任务变化时不需从头设计。

4.2 任务六维多样性空间

维度说明示例
1) Coordination Strategies协调策略类型几何编队(lines/circles/characters)、区域访问(region visitation)、动态目标跟踪、目标驱赶(herding)
2) Motion Patterns运动轨迹类型直线(straight lines)、之字形(zigzag)、螺旋(spiral)
3) Constraints附加约束特定方向进入区域、动作持续时间限制、最大速度、禁止区域
4) Trigger触发条件环境事件触发任务终止/切换（如机器人靠近禁止区域1.5m→立即撤离）
5) Task Finish Condition原子任务完成标准到达指定位置（容差0.1单位内）、驱赶对象进入目标区域范围
6) Mission Finish Condition整体任务完成标准所有子任务完成 OR 高层目标达成（如区域覆盖≥60%）

4.3 Task Type

这是一个LLM-机器人接口层的研究——不是底层控制，也不是纯上层任务规划，而是连接自然语言意图与多机器人协调执行的"中间件"。核心贡献是设计了一套通用框架，使得多机器人协调可以像"对话"一样被指定——这是对传统"一个任务一个算法"范式的基本性颠覆。

5. Challenge — Why Is It Hard?

5.1 Essence (First Principles)

💡 第一性原理诊断
核心矛盾：自然语言的高度模糊性 vs 机器人控制的精确性要求。

（1）语义鸿沟：用户说"形成圆形编队"——LLM理解这句话，但"圆形"在多机器人语境下意味着什么？圆心在哪？半径多大？每台机器人在圆的哪个位置？这些信息在自然语言中天然缺失，LLM必须"猜测"并补全——而猜测就有出错可能。

（2）依赖关系推理：多机器人任务涉及子任务间的复杂时序约束——"A和B并行执行，都完成后C开始；但如果D触发条件满足，A和B立即中止，E启动"。这种拓扑关系的推理需要LLM不仅理解任务内容，还要理解任务间的因果/时序依赖——这是纯文本推理的极限。

（3）代码生成的正确性-灵活性权衡：给LLM过强约束（只能调预定义API）→灵活性受限（螺旋轨迹这种库中没有的行为就无法生成）。给LLM完全自由→可能产生幻觉或物理不可行操作。如何设计"恰到好处的约束"是核心工程挑战。

三者叠加：LAN2CB 需要同时解决语义理解、依赖推理和受约束代码生成三个层级的挑战——任何一层失败都会导致整个流水线崩溃。

5.2 Prior Methods & Their Limits

Prior MethodCore IdeaWhy It Fails for Multi-Robot
Single-Robot LLM 集成（SayCan, Code-as-Policies, ProgPrompt, LM-Nav 等） LLM解释单台机器人的NL指令，生成执行计划或代码 完全无法处理多机器人协调——没有团队层面的任务分配、依赖分析和并行执行管理。单机器人LLM不了解"其他机器人在做什么"
LLM→形式化语言（Lang2LTL, AutoTAMP, NL2TL, LLM+P 等） 将NL转换为LTL/PDDL等形式化规划语言后交给经典规划器求解 受限于任务复杂度——形式化语言无法描述"驱赶+编队+触发切换"这类复杂组合。且多机器人场景的状态空间爆炸使得经典规划器难以求解
LLM 多机器人对话（RoCO, Mandi et al. 2024） 机器人通过 LLM 对话代理进行协调——每台机器人查询LLM并"互相对话" 无法扩展到≥10台机器人——对话式协调的通信和推理开销随团队规模平方增长。且不支持复杂长时序任务
LLM 任务规划器（SMART-LLM, Kannan et al. 2024） LLM驱动的任务分解和分配，但仅限高层次任务级别 缺乏领域知识库（不了解具体机器人能力限制）、泛化性限于in-context示例、不支持运动级指令（如"以螺旋路径访问"）
GenSwarm — NL→多机器人代码（Ji et al. 2025） 将NL任务描述直接翻译为可执行Python的多机器人代码 最接近LAN2CB——但仅聚焦简单单团队场景，无多团队依赖+触发事件。且仅靠in-context learning，缺乏结构化知识库导致泛化受限

5.3 Key Challenges

  • 挑战 1 — 模糊 NL 到精确行为树的映射：用户描述"两组机器人并行工作，如果第一组靠近禁止区域就切换到另一任务"——隐含了分组逻辑、并行拓扑、触发条件、切换目标等多个决策点。LLM必须正确解析所有信息并填入行为树标准字段。模板标准化缓解了这个问题（成功率+15%），但模板本身需专家设计。
  • 挑战 2 — 依赖关系的动态演变：触发条件满足后，任务依赖关系改变——原本"A→B→C"可能变成"B→D→E"。LLM需动态更新依赖分析而非一次性静态规划——这增加了API调用次数和整体成本（Category C平均58K input tokens）。
  • 挑战 3 — 元行为库的覆盖面限制：当前库仅支持 visit_points / follow_targets / herd 三种行为。Category B 任务需 LLM "扩展"这些行为（如用直线合成螺旋轨迹）——本质上等于让 LLM 编写新控制算法，增加出错概率。

6. Method & Key Formulas

6.1 Core Algorithm — LAN2CB 五阶段闭环

LAN2CB 的核心算法包括五个阶段：Mission Analysis（模板标准化+依赖分析+综合行为树生成）、Code Generation（四类代码合成）、Execution（机器人部署）、Progress Update（状态更新与重规划）、Completion（终止检测）。以下拆解关键机制。

Phase 1: 模板标准化

Phase 1 — 模板标准化的作用 $$ \mathcal{D}_{\text{raw}} \xrightarrow{\text{LLM + ExpertTemplate}} \mathcal{D}_{\text{struct}} $$ $$ \Delta_{\text{success}} \approx +15\% \quad \Delta_{\text{error-rate}} \approx -53\% \text{ (2.14× fewer errors)} $$
$\mathcal{D}_{\text{raw}}$ = 用户原始 NL 描述（常省略原子任务定义和执行依赖等关键信息） |  $\mathcal{D}_{\text{struct}}$ = 标准化描述（含 Phase 概览、每 Task 的 task_description/finish_condition/trigger_condition 等完整字段） |  关键量化发现：模板使整体成功率从 78% 提升到 93%，错误频率降低 2.14 倍。模板本质上强制补全了 NL 中天然缺失的信息——这一点在 Category B 中效果最显著（73%→93%，+20%）。

Phase 2: 依赖分析 — 构建任务拓扑图

Phase 2 — 任务依赖关系 $$ \mathcal{G} = (V, E) \quad V = \{t_1,\dots,t_K\} \quad E = \{(t_i, t_j) \mid t_i \text{ must precede } t_j\} $$ $$ \text{Topology} \in \{\text{Sequential}, \text{Parallel}, \text{Trigger-Based}\} $$
$V$ = 原子任务集合（LLM 从标准化描述中提取） |  $E$ = 依赖边——$t_i$ 必须在 $t_j$ 之前完成  |  关键机制：(1) 初次分析时忽略触发相关的任务；(2) 运行时若触发条件激活 → LLM 被重新调用来更新依赖图（移除被触发的旧任务，添加触发后的新任务）。这使得依赖图随任务进展动态演变。

Phase 3: 行为树构造与就绪节点选择

Phase 3 — 行为树节点 $$ \text{ActionNode} = \langle \text{type}, \text{key\_action}, \text{robot\_ids}, \text{points}, \text{trigger\_condition}, \text{finish\_condition}, \text{status} \rangle $$ $$ \text{ReadyToRun} = \{n \in \text{ActionNodes} \mid \forall p \in \text{Predecessors}(n),\ \text{status}(p) = \text{Success}\} $$
关键字段：key_action ∈ {visit_points, follow_targets, herd}, robot_ids（该节点控制的机器人列表），trigger_condition（触发条件，NL描述），finish_condition（完成条件），status ∈ {Idle, Running, Success, Failure}  |  就绪条件：Action节点只有当所有前置依赖节点都处于 Success 状态时才被选中执行——保证任务不会"抢跑"。

Phase 4: 四类代码生成

对每个就绪的 Action 节点，LLM 生成以下四段 Python 代码：

  • (a) Robot Execution Code：Goal Generation（选择行为）→ Goal Allocation（分配目标到机器人）→ Motion Generation（生成路径）的三步流水线，全部基于元行为库 API
  • (b) Trigger Condition Code：函数签名 trigger_condition_generation(robot_positions, object_positions, ...) -> Dict[int, bool]——检查每个 Action 节点的触发条件是否满足
  • (c) Action Node Finish Condition Code：检查该节点所有机器人是否都满足完成条件（如所有机器人到达点位容差0.1内）
  • (d) Mission Finish Condition Code：生成一次——检查全局任务是否完成（所有子任务成功 OR 覆盖率60%等高层目标）

Phase 5: 执行与动态重规划循环

Phase 5 — 执行循环逻辑 $$ \textbf{while}\ \text{not MissionFinish}: $$ $$ \quad \textbf{if}\ \exists n \in \text{Active}:\ \text{TriggerCondition}(n) = \text{True}: $$ $$ \qquad \text{LLM\_UpdateDependencyAnalysis}() \rightarrow \text{LLM\_NewBehaviorTree}() $$ $$ \quad \textbf{if}\ \text{FinishCondition}(\text{ActionNode } n) = \text{True}: $$ $$ \qquad \text{LLM\_UpdateBTStatus}(n, \text{Success}) \rightarrow \text{LLM\_SelectNextReady}() $$
触发→重规划：移除被触发的当前任务 → 添加触发后新任务 → 重新生成整个行为树 → 选择新就绪节点 → 生成新代码  |  完成→状态更新：更新Action节点状态为Success → 级联检查父节点（若父节点完成条件仅依赖已成功的子节点）→ 选择下一就绪Action节点 → 继续  |  复合节点完成：若Parallel/Sequence的完成条件满足 → 将所有未执行/运行的子节点标记为Failure → 复合节点标记为Success。

7. Fundamental Principles — 核心技术原理详解

理解 LAN2CB 需要掌握三个核心原理：行为树的任务表示能力、LLM-in-the-loop 的闭环决策、以及约束式代码生成的抽象层次设计。

7.1 行为树原理 — 为什么选行为树而非 FSM 或 LTL

原理 1 — 行为树的形式化 $$ \text{BT} = \langle \text{Root}, \{\text{Composite}\}, \{\text{Action}\} \rangle $$ $$ \text{Composite} \in \{\text{Sequence}, \text{Parallel}\} $$ $$ \text{Status} \in \{\text{Idle}, \text{Running}, \text{Success}, \text{Failure}\} $$
Sequence：从左到右依次执行子节点——任一 Failure→全部 Failure；全部 Success→Sequence Success。天然表达"先A后B再C"的顺序依赖。 |  Parallel：所有子节点同时执行——可配置成功策略（any/all）。天然表达"A组和B组并行"的并行依赖。 |  选择行为树的三个理由：(1) 树状结构+JSON表示与LLM的上下文理解能力天然匹配；(2) 四状态管理（Idle/Running/Success/Failure）使触发和重规划的状态追踪非常直观；(3) 模块化可组合性——触发条件满足时只需替换受影响的子树，而非重建整棵树。

7.2 LLM-in-the-loop 原理 — LLM 作为持续决策者

原理 2 — 闭环控制中的 LLM 角色 $$ \text{state}_{t+1} = \text{LLM}(\text{state}_t, \text{observation}_t, \text{BT}_t) $$ $$ \text{code}_{t+1} = \text{CodeGen}(\text{BT}_{t+1}, \text{ReadyNodes}_{t+1}) $$
与传统 LLM 规划（SayCan, LLM-Planner）的本质区别：传统方法假设任务可一次性完整规划——但多机器人场景中环境变化是常态。LAN2CB 将 LLM 置于执行循环中——每次触发/完成条件满足时，LLM 重新根据最新环境和行为树状态做出决策。这赋予了系统反应性（reactivity）——多机器人协调从模拟走向真实的关键跨越。 |  代价：多次 LLM 调用增加 tokens 消耗和延迟——Category C 任务平均调用次数更多（tokens: 58K input / 15K output）。

7.3 约束式代码生成原理 — 灵活性 vs 安全性的平衡

原理 3 — 约束式代码生成的抽象层次 $$ \text{LLM} \not\rightarrow \text{Raw Python} \quad \text{（过度自由 → 幻觉）} $$ $$ \text{LLM} \rightarrow \text{API Compositions} \quad \text{（适度约束 → 安全）} $$ $$ \text{API} = \{\text{visit\_points}, \text{follow\_targets}, \text{herd}, \text{default\_alloc}, \text{min\_conflict\_alloc}, \text{motion\_gen}\} $$
为什么禁止自由代码生成？——Huang et al. 2024 的实验证明无约束 LLM 代码生成在机器人场景中易产生：(1) 幻觉——调用不存在的函数；(2) 逻辑错误——物理不可行的操作序列；(3) 语法错误——格式不正确的 Python。 |  约束式方案：将核心功能封装为验证过的 API，LLM 只决定调用哪个函数、什么参数、什么顺序——相当于将"自由代码生成"降级为"有界参数选择+胶水逻辑"。Category B 中 LLM 合成的螺旋轨迹也是基于直线运动 API 的"扩展组合"而非从零编写。
💡 三个原理的系统级协同
行为树提供结构化任务表示（依赖可追溯+状态可管理）→ LLM-in-the-loop 提供闭环动态决策（环境变化时有反应）→ 元行为库提供安全代码生成（避免幻觉）。三者形成"表示-决策-执行"的完整闭环，缺一不可。这也解释了为什么 LAN2CB 相比 SMART-LLM（有决策无安全执行）、RoCO（有对话无双层架构）、AutoTAMP（有表示无动态重规划）有明显优势。

9. Potential Flaw & Extensions

9.1 Limitations

  • 假设前提：(1) 需要稳定 LLM API 连接——网络中断会导致机器人"卡住"（无法获得新指令）；(2) 依赖专家模板和元行为库——仍需要一次性人工成本；(3) 任务复杂度有上限——Category C 87% 成功率意味更复杂真实任务失败率更高。
  • 适用场景：目前仅支持地面机器人的 visit_points/follow_targets/herd 三种行为。团队规模限于 10 台以内——更大规模可能超出 LLM 上下文窗口和推理能力。
  • 可扩展性：(1) 每新增行为需人工编写对应的库函数——限制自主进化能力；(2) LLM 调用次数随任务复杂度线性增长——Category C 平均 $0.24/任务，长时间运行累积成本高。

9.2 Extension Directions

  • 维度扩展：增加 UAV 空中行为（mapping/inspection/perching）—支持混合空地团队协调。需相应扩展 Goal Generation 和 Motion Generation 库
  • 条件放松：(1) RAG 集成——作者提出用 Retrieval-Augmented Generation 利用历史任务数据减少对专家模板依赖；(2) 本地 LLM 部署——如 PRISM 蒸馏到本地模型，消除云 API 依赖，提高通信中断鲁棒性
  • 约束加入：(1) 安全约束——在元行为库中加入物理安全验证；(2) 机器人间直接通信——当前机器人仅通过 LLM 间接协调，加入 P2P 通信可提升效率

9.4 Paper-Worthy Deep-Dive

LAN2CB + PRISM 集成——"完全本地自主的多机器人协调"。若将 LAN2CB（LLM驱动多机器人协调）和 PRISM（LLM蒸馏到本地SLM）结合，将实现：(1) 用户用NL指定任务；(2) 全部推理在本地GPU（Jetson Orin级）运行；(3) 零网络依赖的多机器人自主协调。核心挑战：(a) 蒸馏后SLM能否保持LAN2CB的依赖分析精度；(b) 代码生成在SLM上是否可靠；(c) 多机器人协调的上下文长度是否在SLM窗口范围内。

10. Motivation — First-Principles Derivation

10.1 Idea Derivation Chain

  1. 问题的本质是什么？
    多机器人协调需要从 NL 任务描述到机器人可执行代码的转换。传统方法将此分解为多个子问题（建模→算法→代码），每步需领域专家手动完成。任务描述改变时整个流程需重做——而真实世界中任务变化是常态。
  2. 前人如何处理？为什么不够？
    四条路线：(i) 经典规划——每个场景定制算法，无泛化性；(ii) 形式化方法（LTL/PDDL）——表达力有限；(iii) 单机器人 LLM——无法处理多机器人协调；(iv) 多机器人 LLM（RoCO/SMART-LLM）——无法扩展到大规模团队或缺乏领域知识。
  3. 如果让 LLM 扮演"系统集成者"呢？
    LLM 优势在于理解 NL 和生成结构化输出。将多机器人协调所需的"原子能力"（目标生成/分配/运动规划）封装为 API——LLM 工作从"编写所有代码"变为"编排 API 调用"——大幅降低出错概率。LLM 的 NL 理解能力直接处理模糊任务描述（"形成漂亮的编队"）。
  4. 还不够——需要处理"变化"。
    真实世界任务非静态——触发条件、完成条件、环境变化都导致任务结构改变。LAN2CB 将 LLM 置于"闭环"中——每次状态变化调用 LLM 重新分析——赋予系统"反应性"——这是传统规划方法（经典或LLM驱动）所缺乏的。
  5. 最简单的方案
    用户一句话→LLM解析为行为树→LLM生成代码→机器人执行→发现问题→LLM修改行为树→生成新代码→继续。这个方案之所以新颖，是因为它将传统上分开的"任务规划"和"运行时重规划"统一到同一个 LLM 框架中。

10.2 One-Sentence Motivation

💡 一句话动机
传统多机器人协调需要专家手动翻译 NL→数学建模→算法设计→代码编写——劳动密集、不可扩展、缺乏灵活性——那为什么不直接让大语言模型充当"通用任务解析器+代码生成器+动态重规划器"，把多机器人协调变成一个"说话就能指挥"的系统？

10.3 How You'd Arrive at This Idea

  • Step 1 — 观察：多机器人协调的"可复用性困境"——每新任务（即使微小不同）必须重新设计流水线。学术界有成熟单机器人LLM工作（SayCan等），但在多机器人场景几乎无用。
  • Step 2 — 分解：NL→多机器人代码 = (1) 任务理解（NL→结构化表示）；(2) 代码生成（表示→可执行代码）；(3) 动态适应（发现问题→修改表示和代码）。
  • Step 3 — 匹配工具：(1) 行为树——天然适合多机器人层次结构+依赖关系；(2) 元行为库——机器人能力封装为可复用API；(3) LLM-in-the-loop——执行循环中持续调用LLM更新状态。
  • Step 4 — 组装：LAN2CB = Mission Analysis（行为树）+ Code Generation（元行为库）+ LLM-in-the-loop（动态重规划）+ 模板标准化（关键增强，+15%）。

11. Key Takeaways

11.1 Lessons Learned

  • 方法论层面："约束式LLM应用"——不是让LLM自由发挥，而是提供结构化接口（行为树 schema + API库），将LLM从"创造者"变为"编排者"。对于UAV领域启示：不要试图让LLM直接生成飞行控制代码——让它在高层次编排已验证的行为原语。
  • 技术技巧：(1) 模板的力量——仅标准化NL描述就能+15%成功率——结构化提示在机器人领域至关重要；(2) 行为树的四状态管理+级联更新——处理动态任务切换的优雅方案；(3) 模块化分离——Mission Analysis/Code Generation 可独立优化替换；(4) 消融实验设计——通过开/关模板量化组件贡献——评估LLM系统的标准方法。
  • 写作层面：9任务分3类（A/B/C）系统性覆盖任务空间——"复杂度递进"让读者清楚看到能力边界——哪类完美（A）、哪类仍有挑战（C）。

11.2 Directly Applicable to My Research

  • UAV 编队控制：LAN2CB 的"Group 1 形成圆形"可直接启发"形成钻石形V形"的 UAV 编队。需扩展元行为库加入 altitude change / loiter / dynamic obstacle avoidance 等空中行为
  • 空地协作：结合 Paper 10011（Cladera et al.）的空地系统——用 NL 指定空地协作任务——极具前景的研究方向
  • 视觉伺服：元行为库加入 approach_target_visually / track_with_camera 等视觉行为——可实现"飞到目标上方保持视觉锁定"等基于视觉的协调任务

Code-as-Policies (Liang et al., 2023)"用 LLM 编排 API 调用而非生成原始代码"的思想源头——LAN2CB 元行为库设计直接受此启发
SMART-LLM (Kannan et al., 2024)LLM 驱动的多机器人任务规划——LAN2CB 的结构化知识库和元行为库直接解决了其领域知识缺乏和运动级指令不支持的限制
SPINE (Ravichandran et al., 2025)同组论文——LLM驱动导航规划器。LAN2CB 可与之集成扩展单机器人规划到多机器人团队
RoCO (Mandi et al., 2024)LLM 对话式多机器人协调——虽不适用于大规模团队，但"对话式协调"概念可能与 LAN2CB 行为树方法互补
