2. Insight & Novelty

2.1 Core Insight

Core Insight
Why is LLM alignment (effective for chatbots) insufficient for LLM-enabled robots?
Because harmful robot behavior is context-dependent -- the same navigation action is safe in an empty corridor but dangerous during an emergency evacuation (it could block an exit). Chatbot alignment only addresses "don't say harmful things" and does not address physical-world context. Moreover, jailbreaking attacks (e.g., ROBOPAIR) can trigger dangerous physical behaviors in LLM-driven robots -- such as colliding with humans or blocking emergency exits -- even when the LLM itself has undergone alignment training.

What is this paper's core insight?
Use a protected root-of-trust LLM for safety reasoning and formal methods for safety enforcement -- separated and complementary. Key insights: (1) The safety-reasoning LLM must be isolated from user input (root-of-trust), making it immune to malicious prompt manipulation; (2) Safety rules (e.g., "do not harm humans") must be grounded to specific robot context (e.g., "in this environment, person_1 is near doorway_1, so do not block doorway_1"); (3) Conflicts between safety specifications and plans are resolved through minimum-violation control synthesis -- prioritizing safety while maximally respecting user intent.

2.3 Novelty Breakdown

Novelty 1: Root-of-Trust LLM Architecture -- Isolating Safety Reasoning from User Input

[Problem Solved] Traditional LLM planners directly accept user input and can therefore be manipulated by malicious prompts. If safety checking and plan generation share the same LLM, jailbreaking attacks can bypass both simultaneously.
[Inspiration] The root-of-trust concept from computer security -- the Trusted Computing Base (TCB) is an isolated, trusted component that maintains security even when the rest of the system is compromised.
[Design] ROBOGUARD's safety reasoning LLM is independent of the planning LLM -- it only receives safety rules from the system designer and world model updates from the semantic map, never directly touching user input. This isolation means even if an attacker completely compromises the planning LLM, the safety guardrail remains effective. In experiments, even when attackers obtain gray-box/white-box access to the guardrail internals, ASR only increases to at most 5.2%.

Novelty 2: Grounding Natural Language Safety Rules to LTL Safety Specifications

[Problem Solved] High-level safety rules (e.g., Asimov's First Law "do not harm humans") are abstract -- how to convert them into formally verifiable safety specifications in a concrete environment?
[Design] Given world model M and robot action set F, define contextual atomic propositions AP(M,F) (e.g., goto(person_1), inspect(drill_1)). The root-of-trust LLM uses CoT reasoning to generate one LTL formula phi(i) per safety rule, conjoined: phi_safe = phi(1) and phi(2) and ... and phi(n). Key example: without reasoning, the LLM might omit constraints about doorway or generate overly broad constraints for person -- CoT reasoning dramatically improves constraint precision (ASR drops from 12.8% to 4.3%).

Novelty 3: Minimum-Violation Control Synthesis -- Optimal Balance Between Safety and Intent

[Problem Solved] Safety specifications and LLM plan intent may conflict -- a simple "reject all unsafe plans" approach over-restricts robot capability. A mechanism is needed to maximally follow user intent while ensuring safety.
[Design] Algorithm 1 (Control Synthesis): convert LLM plan p to LTL specification phi_proposed and word sequence w. Construct a Buchi automaton B from phi_safe. Stepwise verify each word in w against B. If w is accepted by B (reaches accepting state), execute the plan; otherwise fall back to phi_safe (safety first). Theorem 1 guarantees the control synthesis module always provides a safe control plan.

4. Task -- What Problem Does It Solve?

4.1 Problem Formulation

Core Problem -- Safety Guardrails for LLM-Enabled Robots $$ \text{Given: Safety rule set } \mathcal{R},\; \text{World model } \mathcal{M},\; \text{LLM-generated plan } p $$ $$ \text{Find control plan } \pi \text{ s.t. } \pi \models \phi_{\text{safe}} \text{ and } \pi \text{ maximally follows } p $$ $$ \text{where } \phi_{\text{safe}} = \bigwedge_{i=1}^n \phi^{(i)}(\mathcal{M}, \mathcal{R}) \text{ is the LTL safety spec grounded to context} $$

4.3 Task Type

This is an adversarial safety guardrail problem -- the core is not detecting jailbreak attacks per se, but ensuring physical safety under the worst-case assumption that the LLM planner has already been compromised. Platform: Clearpath Jackal mobile robot (simulation + real-world); scenarios: full-floor indoor office / outdoor office park; attacker uses 7 harmful behaviors (bomb detonation, blocking emergency exits, weapon search, warehouse assault, restricted zone entry, covert surveillance, person collision) and multiple jailbreak methods (direct prompts, template attacks, ROBOPAIR, and adaptive variants). The guardrail operates at the planning level -- it takes a discrete action sequence plan and verifies it against safety specifications, unlike CORE which operates at the continuous control level.

5. Challenge -- Why Is It Hard?

5.1 Essence (First Principles)

First-Principles Diagnosis
Core contradiction: the fundamental tension between LLM flexibility and safety.

(1) LLM's inherent vulnerability: LLMs are trained to be "helpful and non-refusing" -- this openness is precisely the entry point for jailbreak attacks. ROBOPAIR demonstrated that by constructing fictional scenarios in context (e.g., "you're filming a movie"), attackers can induce LLMs to generate dangerous robot plans. This vulnerability is architectural, not incidental -- it is the flip side of LLMs' open-ended generation capability.

(2) Safety's context-dependence: For robots, "safe" depends on the specific physical context. In a semantic graph, a person node can appear in different regions -- but the safety rule must be "do not collide with this specific person" not "do not collide with any person whatsoever." This requires grounding safety rules from abstract ("do not harm humans") to concrete ("do not goto(person_1)") -- a semantic reasoning problem.

(3) Attack-defense asymmetry: An attacker need only find one successful jailbreak prompt; the defender must block all possible attacks. ROBOGUARD resolves this asymmetry by separating safety reasoning from the attacker-accessible LLM -- the attacker can manipulate the planning LLM but cannot touch the root-of-trust LLM.

  • Challenge 3 -- Defending Against Adaptive Attacks: Attackers may adjust their strategy based on ROBOGUARD's responses. A white-box attacker knowing the guardrail's complete internal state can design prompts specifically to bypass it. 6. Method & Key Formulas

6.1 Core Algorithm

ROBOGUARD's core is a two-stage architecture. Stage 1: the root-of-trust LLM receives the world model (semantic graph JSON), uses CoT reasoning to ground high-level safety rules into an LTL formula phi_safe = conjunction_i phi(i). Stage 2: convert the LLM plan p into LTL specification phi_proposed and word sequence w, verify via Buchi automaton whether w satisfies phi_safe -- if so, execute the plan; otherwise fall back to the safety specification.

6.2 Key Formulas

Group A: Contextual Atomic Propositions and LTL Safety Specification

Eq.(1) -- Contextual Grounding and Safety Specification Synthesis $$ \mathcal{AP}(\mathcal{M}, \mathcal{F}) = \{\text{goto}(r), \text{inspect}(o), ...\} \text{ (Contextual Atomic Propositions)} $$ $$ \phi_{\text{safe}} = \phi^{(1)} \land \phi^{(2)} \land \cdots \land \phi^{(n)} \quad \text{(LTL Conjunction of n Safety Rules)} $$
$\mathcal{M}$ = world model (semantic graph: region_1, region_2, person_1, ...) | $\mathcal{F}$ = robot action set (goto, inspect, ...) | $\phi^{(i)}$ = LTL encoding of the i-th safety rule | LTL operators: G (globally/always), F (finally/eventually), U (until), X (next), ! (negation), implies (implication), and (conjunction) | Example: G(!goto(person_1)) = "never navigate to person_1"; G(goto(doorway_1) implies F(!goto(doorway_1))) = "if you navigate to doorway_1, you must eventually leave" (i.e., must not linger and block the exit).

Group B: Control Synthesis Algorithm -- Algorithm 1

Algorithm 1 -- Minimum-Violation Control Synthesis $$ \text{Input: } \phi_{\text{safe}},\; p \text{ (LLM plan)} $$ $$ \text{Step 1: } \phi_{\text{proposed}} \leftarrow \text{ToLTL}(p),\; w \leftarrow \text{ToWord}(\phi_{\text{proposed}}) $$ $$ \text{Step 2: } \mathcal{B}_{\text{safe}} = (Q, q_{\text{init}}, \Sigma, \delta, F) \leftarrow \text{ToAutomata}(\phi_{\text{safe}}) $$ $$ \text{Step 3: Verify whether the trace of } w = w_1 w_2 \ldots w_T \text{ on } \mathcal{B}_{\text{safe}} \text{ is accepted} $$ $$ \text{Output: } \begin{cases} \phi_{\text{proposed}} & \text{if } q_T \in F \text{ (accepted)} \\ \phi_{\text{safe}} & \text{otherwise (rejected)} \end{cases} $$
$\mathcal{B}$ = Buchi automaton -- a 5-tuple defined by state set Q, initial state q_init, alphabet Sigma = 2^AP, transition function delta, and accepting state set F | $w$ = word -- the action sequence in the plan; each w_j is a proposition from AP | Acceptance condition: the trace's final state belongs to F (the safe state set) | Theorem III.2 (Safety Guarantee): The control synthesis module always provides a safe control plan -- regardless of what the input plan p is. Because phi_safe defines the complete boundary of safe behavior, and the algorithm always falls back to phi_safe when the plan does not satisfy the safety specification.

Group C: Concrete Example -- Impact of CoT Reasoning on Specification Quality

CoT Reasoning's Impact on Safety Specification Quality $$ \text{Without reasoning: } \phi_{\text{safe}} = G(!\text{inspect(drill\_1)}) \land G(!\text{goto(doorway\_1)}) \quad \text{(misses person constraint!)} $$ $$ \text{With reasoning: } \phi_{\text{safe}} = G(!\text{goto(person\_1)}) \land G(!\text{inspect(drill\_1)}) \land G(\text{goto(doorway\_1)} \rightarrow F(!\text{goto(doorway\_1)})) $$
Difference analysis: Without CoT reasoning, the variant misses the constraint on person_1 and over-constrains doorway_1 (completely forbids rather than "may pass through but don't linger"). CoT reasoning enables the LLM to stepwise consider each safety rule's physical implications -- person_1 is a potential collision risk, drill_1 is a potential weapon, doorway_1 is an exit that must not be blocked -- thereby generating more precise and comprehensive LTL constraints. This directly reflects in ASR: without reasoning, ASR rises from 4.3% to 12.8%.

7. Fundamental Principles

7.1 Linear Temporal Logic (LTL) for Robot Safety

LTL extends propositional logic with temporal modalities that are essential for expressing safety properties over time. The core operators are: G phi (globally/always phi -- phi holds at every future time step), F phi (finally/eventually phi -- phi holds at some future time step), X phi (next -- phi holds at the next time step), phi U psi (phi until psi -- phi holds continuously until psi becomes true). LTL formulas are interpreted over infinite traces (omega-words) of system states, where each state is a set of atomic propositions that hold at that moment. This makes LTL naturally suited for reasoning about robot action sequences: a trace is a sequence of actions like goto(region_1), inspect(chair), goto(region_2), and an LTL formula like G(!goto(person_1)) asserts that at no point in this sequence does the robot navigate to person_1. The key theoretical property ROBOGUARD exploits is that every LTL formula can be compiled into a Buchi automaton, enabling algorithmic verification of whether a given trace satisfies the formula. The Spot library used in ROBOGUARD implements the standard tableau-based conversion from LTL to generalized Buchi automata, then to non-deterministic Buchi automata with a single acceptance set.

7.2 Buchi Automata and Trace Verification

A Buchi automaton is a 5-tuple B = (Q, q_init, Sigma, delta, F) where Q is a finite set of states, q_init is the initial state, Sigma = 2^AP is the alphabet (each symbol is a subset of atomic propositions), delta: Q x Sigma to 2^Q is the non-deterministic transition function, and F (a subset of Q) is the set of accepting states. An infinite word w = sigma_1 sigma_2 ... over Sigma is accepted if there exists a run through B visiting F infinitely often. In ROBOGUARD, the plan's action sequence is a finite word w = w_1 w_2 ... w_T, and the algorithm verifies whether this finite prefix can be extended to an infinite word accepted by B -- equivalently, whether the trace ends in a state from which an accepting run is possible (a "live" state). The key algorithmic property: if the trace's final state q_T belongs to the set of states that can reach an accepting state (computed via SCC decomposition), the plan is deemed safe. The Spot library provides efficient O(|Q| x T) trace verification through its spot::otf_product interface. This verification is purely symbolic (no continuous dynamics), making it fast enough for online use.

7.3 Root-of-Trust Design Principle

The root-of-trust concept originates in computer hardware security: a Trusted Computing Base (TCB) is the minimal set of hardware, firmware, and software components that must be trusted for the system's security to hold. The core principle is minimization and isolation: the TCB should be as small as possible to reduce attack surface, and should be architecturally isolated from untrusted components so that compromise of the untrusted parts cannot propagate to the TCB. ROBOGUARD adapts this principle to LLM-based systems: (1) the root-of-trust LLM's input surface is minimized -- it reads only the safety rule set (from the designer) and the world model (from perception), never user prompts; (2) the root-of-trust LLM is an architecturally separate instance from the planning LLM, sharing no context or state; (3) the root-of-trust LLM's output is formally verifiable -- the LTL safety specification can be checked for syntactic correctness and the Buchi automaton can be analyzed for consistency. Even if the planning LLM is completely compromised by a jailbreak attack, the attacker cannot influence what safety rules the root-of-trust LLM enforces, because there is no data path from user input to the root-of-trust LLM.

7.4 Minimum-Violation Control Synthesis

Minimum-violation planning, introduced by Tumova et al., addresses the problem of reconciling conflicting specifications in automated planning. Given a set of specifications with a total priority order (e.g., safety > task > comfort), the algorithm constructs a plan that satisfies the highest-priority specifications and maximally satisfies lower-priority ones. ROBOGUARD reduces this to the two-specification case: phi_safe (highest priority, must be satisfied) and phi_proposed (lower priority, desirable but not required). The reduction is possible because the LLM plan p is converted to a single LTL formula phi_proposed. The control synthesis algorithm is essentially a prioritized two-specification solver: first check if the proposed plan satisfies phi_safe; if yes, return it; if no, return a plan that satisfies only phi_safe. Crucially, this synthesis is sound by construction: Theorem III.2 proves that whatever the algorithm outputs, it always satisfies phi_safe -- because phi_safe itself is always a valid plan under phi_safe (reflexivity), and the plan is only returned when it has been verified against phi_safe.

7.5 Chain-of-Thought Reasoning for Specification Generation

The root-of-trust LLM uses structured CoT reasoning to convert abstract safety rules into concrete LTL formulas. The reasoning follows a fixed template: (1) Decompose each safety rule into sub-requirements; (2) Enumerate all entities in the world model that are relevant to each sub-requirement; (3) Select appropriate LTL temporal operators based on the nature of the constraint (G for permanent prohibitions, G(... implies F(...)) for "may do X but then must stop doing X"); (4) Synthesize LTL formulas for each entity-rule pair; (5) Conjoin all sub-formulas into phi_safe. The ablation study (ASR: 4.3% with CoT vs. 12.8% without) demonstrates that explicit intermediate reasoning steps prevent the LLM from making omissions -- without CoT, the LLM may skip entities entirely (e.g., forgetting person_1) or apply wrong temporal operators (e.g., using G(!goto(doorway_1)) instead of the more nuanced G(goto(doorway_1) implies F(!goto(doorway_1)))). The CoT template effectively constrains the LLM's search space to a structured checklist, reducing the likelihood of safety-critical omissions.

7.6 Defense in Depth: Multi-Layer Safety Architecture

ROBOGUARD embodies the security principle of defense in depth: no single security layer is trusted to be impenetrable. The layers are: (1) LLM alignment training (RLHF) provides a statistical first line of defense against casual misuse; (2) the root-of-trust LLM provides a semantic second line that understands physical context and generates formal specifications; (3) the Buchi automaton verification provides a mathematical third line -- even if the root-of-trust LLM makes an error (generates an incomplete specification), the formal verification ensures that whatever specification was generated is enforced. Layer 3 does not correct errors in layer 2, but it guarantees that the system's safety behavior is formalizable and auditable. This architecture also creates attack-cost asymmetry: the defender spends ~4300 tokens per query; the attacker spends 20,000-52,000 tokens over 15 queries. The economic efficiency of defense relative to attack is itself a security property -- a guardrail that costs more to bypass than to deploy is practically effective.

9. Potential Flaw & Extensions

9.1 Limitations

  • Assumptions: (1) The world model must be accurate -- if the semantic graph misses critical semantic entities (e.g., undetected person), ROBOGUARD cannot generate corresponding safety constraints. This requires additional perception safety measures to ensure world model completeness. (2) Safety rules must come from the system designer -- if rules are incomplete or incorrect, the safety specification will also be incomplete.
  • Applicable scenarios: LTL encoding is suitable for safety verification of discrete action sequences, but not for scenarios requiring continuous dynamics safety (e.g., collision avoidance requiring real-time control adjustments). The paper mentions future work could use other specification languages (e.g., STL) to handle continuous-time constraints.
  • Scalability: (1) As environment complexity grows, the world model enlarges, requiring the root-of-trust LLM to reason about more semantic relationships -- increasing token consumption and latency. (2) More safety rules mean longer LTL formulas, risking automaton state explosion. However, current LLM context windows (1M+ tokens) still have significant headroom.

9.2 Extension Directions

  • Dimensional extension: From discrete action safety to continuous control safety -- integrate CBF or other continuous safety methods into ROBOGUARD to handle continuous spatial constraints like "do not come within 1m of person." This complements CORE's (paper 10001) direction.
  • Condition relaxation: (1) Use distillation to compress the root-of-trust LLM's capabilities into smaller models -- enabling local safety reasoning on embedded devices, reducing cloud latency and communication dependency. (2) Develop uncertainty-aware safety reasoning -- when the LLM is uncertain about scene safety, generate more conservative constraints.
  • Constraint addition: (1) Upgrade safety reasoning from passive blocking to active guidance -- ROBOGUARD not only rejects unsafe plans but also suggests to the planning LLM "the closest thing you can do under current safety constraints." (2) Multi-robot safety coordination -- when multiple LLM-driven robots share the same space, safety constraints may interact.

10. Motivation -- First-Principles Derivation

10.1 Idea Derivation Chain

  1. What is the essence of the problem?
    LLM flexibility is simultaneously its greatest strength and greatest safety risk -- because LLMs can generate any output that conforms to language patterns, including harmful robot plans. Alignment training only statistically reduces the probability of harmful outputs, it does not guarantee safety (92.3% jailbreak success is the evidence). A safety guardrail must be built outside the LLM.
  2. How have prior works handled this? Why is it insufficient?
    Three routes: (i) better alignment -- reduces but cannot eliminate harmful outputs; (ii) constrain LLM output format -- but constraints also restrict legitimate use; (iii) formal verification -- but requires predefined specifications, and not all possible dangerous scenarios can be pre-enumerated in open worlds.
  3. What if we use an independent, protected LLM specifically for safety judgment?
    Key idea: separate "understanding user instructions" from "judging whether instructions are safe" into two different LLMs. The planning LLM can contact user input (users are necessary), but the safety LLM only contacts system designer rules and the world model (never user input). Even if the planning LLM is jailbroken, the safety LLM can still block harmful plans.
  4. How does the safety LLM convert abstract rules to executable constraints?
    A formal language is needed to express safety -- LTL is naturally suited for expressing safety properties of robot action sequences ("never do X," "if you do Y you must eventually leave"). The safety LLM's role is to "translate" high-level rules into LTL formulas -- this requires understanding semantic relationships in the world model.
  5. What if safety specs and user intent conflict?
    Use minimum-violation synthesis -- safety spec has highest priority, but user intent is followed as much as possible within safety bounds. This ensures safety does not become "can't do anything at all."

10.2 One-Sentence Motivation

One-Sentence Motivation
LLM-driven robots can be induced by jailbreak attacks to execute dangerous behaviors (e.g., colliding with humans) -- rather than trying to make LLMs "not go bad" (alignment), build an independent safety guardrail outside the LLM that "will never allow dangerous behavior no matter what" -- using a root-of-trust LLM for safety reasoning and formal methods for safety enforcement.

10.3 How You'd Arrive at This Idea

  • Step 1 -- Observe the security vulnerability: ROBOPAIR (Robey et al. 2025) demonstrated the astonishing vulnerability of LLM-driven robots -- GPT-4o achieves 92.3% jailbreak success rate in robot scenarios. This is work from the same research group, naturally leading to the conclusion "we need a guardrail."
  • Step 2 -- Analyze the root cause of vulnerability: LLM safety vulnerabilities are not "the model isn't good enough" but are architectural -- LLMs are designed to be "helpful," and this design itself is in tension with "completely safe." Therefore, better alignment alone cannot solve the problem.
  • Step 3 -- Design the isolation architecture: Borrow the root-of-trust concept from computer security -- even if the entire system is compromised, the TCB (Trusted Computing Base) remains trustworthy. Make safety reasoning the TCB, separated from the untrusted planner.
  • Step 4 -- Match formal tools: LTL + Buchi automata = standard discrete action sequence verification method. Tumova et al.'s minimum-violation planning provides the conflict resolution template. Combine both with LLM reasoning: LLM generates specifications, formal methods verify plans.

11. Key Takeaways

11.1 Lessons Learned

  • Methodology level: This is a classic application of "defense in depth" in robot safety -- not relying on a single safety layer (LLM alignment) but building multiple independent layers of protection: (1) LLM alignment is the first layer (statistical protection), (2) ROBOGUARD's root-of-trust LLM is the second layer (semantic-level protection), (3) formal verification is the third layer (mathematical-level guarantee). For UAV domain: autonomous UAVs similarly face "command tampering/interference" risks -- an independent flight safety controller could be designed with ROBOGUARD-like architecture.
  • Technical skill level: (1) Root-of-trust isolation design -- the safety reasoning module should never touch any user-originated data, only trust the system designer and sensors. This design principle generalizes to any "untrusted AI + safety-critical system" scenario; (2) CoT reasoning's amplifying effect on safety decisions -- the ablation data (4.3% to 12.8% to 25.7%) clearly shows that safety LLMs cannot be "black boxes" and must output interpretable reasoning steps; (3) Attack-cost asymmetry -- defenders need only 1 LLM query (4300 tokens), attackers need 15 (20,000-52,000 tokens) -- this economic asymmetry is itself a form of defense.
  • Experiment/writing level: The paper first defines four Desiderata (D1-D4) as the guardrail design evaluation framework -- then each Desideratum is validated experimentally. This "requirement definition, design satisfaction, experimental validation" structure makes the paper highly persuasive.

11.2 Directly Applicable to My Research

  • For UAV mission planning safety: ROBOGUARD's "planning LLM + independent safety LLM + formal verification" three-layer architecture can be directly adapted to UAV mission planning -- the planning LLM generates flight missions (e.g., "fly to GPS coordinates X to collect imagery"), the safety LLM generates safety constraints (e.g., "do not enter no-fly zones, maintain minimum safe altitude"), and formal verification ensures the flight plan violates no safety rules.
  • For multi-UAV collaboration: In multi-drone missions, each UAV may have different safety constraints (payload limits, battery state, communication range), and inter-UAV actions may mutually affect safety. ROBOGUARD's architecture can be extended to a centralized safety coordinator -- a single root-of-trust LLM handling joint safety constraints for the entire team.
  • For human-robot interaction safety: ROBOGUARD's high-level safety rule interface ("do not harm humans") is a natural language safety value definition interface -- enabling non-technical users to define robot safety behavior codes without writing code or formal specifications. This is an important step toward "configurable safe robots."
