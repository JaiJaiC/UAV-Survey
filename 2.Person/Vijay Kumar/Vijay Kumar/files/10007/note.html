2. Insight & Novelty

2.1 Core Insight

Core Insight
Q: Why is PDDL domain generation a bottleneck for UAV planning, and why do naive LLM approaches fail?
A: PDDL domain authoring requires simultaneously satisfying (a) strict formal syntax, (b) correct predicate/action semantics, and (c) domain-specific physical constraints (e.g., UAV battery, no-fly zones). Naive LLM prompting produces domains with severe syntax errors and semantic inconsistencies -- LLMs lack the structured reasoning needed to map natural language task descriptions to formal planning representations.

Q: What is the core insight of SPAR?
A: Decompose domain generation into a structured, multi-step reasoning pipeline. Rather than asking an LLM to output a complete PDDL domain in one shot, SPAR breaks it down: (1) retrieve the most relevant example domains via RAG from a curated dataset; (2) generate each action one at a time with CoT reasoning that explicitly considers objects, preconditions, and effects; (3) validate syntax after each action and provide LLM-friendly error feedback for iterative correction. This decomposition mirrors how a human PDDL expert would work -- examine similar examples, think about each action's preconditions/effects, and iteratively debug syntax errors.

2.3 Novelty Breakdown

Novelty 1: First multi-domain UAV PDDL dataset with complexity metric

[Problem] No existing dataset provides diverse, systematically categorized PDDL domains specifically for UAV planning tasks. Prior work uses ad-hoc single domains.
[Design] Curated 30 UAV domains (14 simple, 16 complex) with 9 problems per domain (270 total). Introduced a domain complexity metric with 9 weighted components: number of actions, types, predicates, functions, preconditions, effects, interdependency, action coupling, and overall structural complexity. This enables systematic evaluation across difficulty levels.

Novelty 2: RAG + CoT + iterative syntax feedback pipeline for PDDL generation

[Problem] LLMs generate PDDL with high syntax error rates and semantic inconsistencies when prompted naively. Existing code-generation approaches lack domain-specific structure.
[Design] Three-stage pipeline per action: (a) Abstract the action description, embed it via all-mpnet-base-v2, retrieve top-K similar examples by cosine similarity, re-rank with LLM to select the best; (b) CoT prompt guides step-by-step reasoning about objects, preconditions, and effects before writing PDDL; (c) Syntax checker validates output, returns error feedback in natural language, and the LLM retries until correct. A dynamic fluent list tracks predicates across actions to maintain consistency.

Novelty 3: Comprehensive multi-dimensional evaluation framework

[Problem] Prior PDDL generation work evaluates only syntax validity. Semantic correctness -- whether the generated domain actually models the intended task -- is not assessed.
[Design] Four evaluation metrics: (1) Syntax error count; (2) Executability -- can a planner solve problems on the generated domain; (3) Feasibility -- do plans from the generated domain work on the ground-truth domain; (4) Interpretability -- do plans from the ground-truth domain work on the generated domain. Executability captures "can it plan at all," while Feasibility and Interpretability together capture semantic equivalence.

4. Task -- What Problem Does It Solve?

4.1 Problem Formulation

Core Problem -- Automated PDDL Domain Generation $$ \text{Given: Natural language task description } T, \text{ curated example dataset } \mathcal{D} $$ $$ \text{Generate: PDDL domain } \mathcal{P} = (\mathcal{T}, \mathcal{P}r, \mathcal{F}, \mathcal{A}) \text{ s.t. } \mathcal{P} \text{ is syntactically valid and semantically accurate for } T $$ $$ \text{Where: } \mathcal{T} = \text{types}, \mathcal{P}r = \text{predicates}, \mathcal{F} = \text{functions}, \mathcal{A} = \{a_1, ..., a_K\} \text{ (actions)} $$
$\mathcal{T}$ = PDDL types (e.g., uav, location, package)  |  $\mathcal{P}r$ = predicates with typed parameters (e.g., (at ?uav - uav ?loc - location))  |  $\mathcal{F}$ = numeric functions for resources (e.g., battery-level, fuel)  |  $\mathcal{A}$ = action schemas with :parameters, :precondition, :effect  |  Core Challenge: LLMs must simultaneously satisfy formal syntax constraints AND faithfully capture the semantics of the UAV task described in natural language.

4.3 Task Type

This is an automated knowledge engineering task for robot planning. The core challenge is translating informal natural language descriptions of UAV missions into formal PDDL domain specifications that can be consumed by classical planners. The task sits at the intersection of natural language understanding (extracting types, actions, constraints from text), code generation (producing syntactically valid PDDL), and domain modeling (ensuring semantic faithfulness to the intended UAV task). The output is consumed downstream by PDDL planners to generate executable UAV mission plans.

5. Challenge -- Why Is It Hard?

5.1 Essence (First Principles)

First-Principles Diagnosis
Core contradiction: PDDL is a formal language with rigid syntax and precise execution semantics, but task descriptions are informal natural language with ambiguity and implicit assumptions.

(1) The Syntax-Semantics Gap: PDDL requires exact specification of typed parameters, predicate arities, and Lisp-like S-expression syntax. A single misplaced parenthesis or type mismatch breaks the entire domain. LLMs, trained on natural language, are inherently error-prone on such formal syntax.

(2) The Domain Coupling Problem: Actions in a PDDL domain are not independent -- they share predicates, types, and numeric fluents. If action-A uses (at ?uav ?loc) as an effect and action-B uses it as a precondition, they must agree on parameter types and predicate names. Generating actions independently breaks these couplings.

(3) The Complexity Scaling Problem: As domains grow more complex (more actions, more predicate interdependencies, numeric fluents with complex arithmetic), the LLM's error rate increases super-linearly. Simple domains (2-3 actions) may be generated correctly, but complex domains (6-8 actions with coupled preconditions/effects) far exceed the LLM's single-pass reasoning capacity.

5.3 Key Challenges

  • Challenge 1 -- Retrieving structurally relevant examples: Simple text similarity between task descriptions does not guarantee structural similarity between PDDL domains. Two UAV tasks described similarly may require very different PDDL structures. SPAR addresses this by embedding abstract action descriptions rather than raw task text, and using LLM re-ranking to select the best structural match.
  • Challenge 2 -- Maintaining cross-action consistency: When generating action-3, the LLM must know which predicates and types were defined in actions 1-2. SPAR's dynamic fluent list explicitly tracks all defined predicates/types across actions and injects them into the CoT prompt for each subsequent action.
  • Challenge 3 -- Iterative syntax correction without semantic drift: When the syntax checker rejects an action and the LLM retries, the correction may alter the action's semantics (e.g., changing a precondition to pass syntax but losing the intended meaning). SPAR mitigates this by keeping the CoT reasoning context across retries, anchoring the LLM to the original semantic intent.

6. Method & Key Formulas

6.1 Core Algorithm

SPAR is a three-stage retrieval-augmented generation pipeline. Stage 1 (RAG Retrieval) embeds abstract action descriptions and retrieves the most structurally similar domain from the curated dataset using cosine similarity + LLM re-ranking. Stage 2 (Action-by-Action CoT Generation) generates each PDDL action schema sequentially, guided by the retrieved example and a dynamic fluent list that tracks cross-action predicate/type consistency. Stage 3 (Iterative Validation) checks each action's syntax and provides natural language error feedback for retry until the action is syntactically valid.

6.2 Key Formulas

Group A: Domain Complexity Metric

Eq.(1) -- Domain Complexity Score $$ C(\mathcal{P}) = \sum_{i=1}^{9} w_i \cdot c_i(\mathcal{P}) $$
$c_1$ = number of actions  |  $c_2$ = number of types  |  $c_3$ = number of predicates  |  $c_4$ = number of numeric functions  |  $c_5$ = average precondition size  |  $c_6$ = average effect size  |  $c_7$ = predicate interdependency (shared predicates across actions)  |  $c_8$ = action coupling (actions that must be sequenced)  |  $c_9$ = overall structural complexity (depth of nested expressions)  |  $w_i$ = normalized weights (sum to 1). Simple domains: $C < 0.4$; Complex domains: $C \ge 0.4$.

Group B: RAG Retrieval via Cosine Similarity

Eq.(2) -- Example Retrieval $$ \mathbf{e}_q = \text{Embed}(d_q) \quad \text{(abstract action description embedding)} $$ $$ \text{sim}(\mathbf{e}_q, \mathbf{e}_i) = \frac{\mathbf{e}_q \cdot \mathbf{e}_i}{\|\mathbf{e}_q\| \|\mathbf{e}_i\|} \quad \text{(cosine similarity)} $$ $$ i^* = \arg\max_{i \in \text{Top-K}} \text{LLM\_Rank}(d_q, d_i) \quad \text{(LLM re-ranking)} $$
$d_q$ = abstract action description extracted from user's NL task  |  $\text{Embed}(\cdot)$ = all-mpnet-base-v2 sentence transformer (768-dim)  |  $\mathbf{e}_i$ = pre-computed embedding of the i-th domain in the UAV PDDL dataset  |  Top-K = K=5 candidates by cosine similarity  |  LLM\_Rank = LLM compares query and each candidate, selects most structurally similar. This two-stage retrieval (dense embedding + LLM re-rank) balances efficiency and accuracy.

Group C: CoT Prompting for Action Generation

Eq.(3) -- CoT Reasoning Steps per Action $$ \text{Q1: What objects/types are involved?} \rightarrow \text{:parameters list} $$ $$ \text{Q2: What must be true before this action?} \rightarrow \text{:precondition} $$ $$ \text{Q3: What changes after this action?} \rightarrow \text{:effect} $$ $$ \text{Q4: Write the complete PDDL action schema} $$
Design rationale: This decomposition breaks the complex task of PDDL action authoring into four simpler sub-tasks that align with the PDDL action schema structure. Each sub-task produces intermediate reasoning that is consumed by the next step. The retrieved example provides structural guidance throughout.

Group D: Iterative Syntax Correction

Algorithm 1 -- Action Validation with Retry $$ \text{for each action } a_k: $$ $$ \quad \text{repeat: } a_k \leftarrow \text{LLM}(prompt_{CoT} + fluent\_list + retrieved\_example) $$ $$ \quad \quad \text{result} \leftarrow \text{SyntaxCheck}(a_k) $$ $$ \quad \quad \text{if result is error: } prompt_{CoT} \mathrel{+}= \text{NL\_ErrorFeedback}(result) $$ $$ \quad \text{until syntax valid or max\_retries exceeded} $$
NL\_ErrorFeedback = PDDL parser error translated to natural language (e.g., "Type mismatch: parameter ?uav is declared as type 'uav' but used as 'robot' in precondition line 3"). This LLM-friendly feedback enables effective self-correction.  |  max\_retries = 3 (most actions pass within 1-2 retries).

7. Fundamental Principles

Understanding SPAR requires grasping three foundational concepts: PDDL as a formal planning language, retrieval-augmented generation for structured output, and why decomposition beats monolithic generation for formal languages.

7.1 PDDL Formal Structure -- Why LLMs Struggle With It

Principle 1 -- PDDL as Typed First-Order Logic $$ \text{State } s = \text{set of grounded atoms } \{p(o_1, ..., o_n)\} \cup \{f = v\} $$ $$ \text{Action } a = (\text{name}, \text{:parameters } \vec{x}:\vec{\tau}, \text{:precondition } \phi, \text{:effect } \psi) $$ $$ s' = (s \setminus \text{del}(a)) \cup \text{add}(a) \quad \text{(STRIPS semantics)} $$
Why LLMs struggle: PDDL requires: (a) all variables must be typed and bound in :parameters; (b) all predicates in :precondition/:effect must use declared predicates with correct arities; (c) numeric fluents require consistent arithmetic expressions. These formal constraints have no analog in the statistical patterns LLMs learn from web text. Natural language has no type system, no formal preconditions/postconditions, and no compositional semantics.

The fundamental tension: PDDL requires compositional correctness -- every local sub-expression must be valid for the whole to be valid. LLMs generate text autoregressively -- each token is predicted given previous tokens, but there is no guarantee that token-t+100 will be consistent with token-t. This is why one-shot generation fails: by the time the LLM writes action-5's preconditions, it may have "forgotten" the predicate signatures it defined in action-1.

7.2 RAG -- Why Relevant Examples Are Essential

Principle 2 -- Retrieval as Structural Anchoring $$ P(a_k \mid T, \mathcal{D}) = \sum_{d \in \mathcal{D}} P(a_k \mid T, d) \cdot P(d \mid T) \quad \text{(marginalization)} $$ $$ \approx P(a_k \mid T, d^*) \text{ where } d^* = \arg\max_{d} \text{sim}(T, d) \quad \text{(top-1 approximation)} $$
$P(a_k \mid T, d)$ = probability of generating action k correctly given task T and example d  |  Intuition: The retrieved example d* provides a structural prior -- it shows the LLM what a valid PDDL domain for a similar task looks like. This anchors the generation to a correct structure template, reducing the search space from "all possible PDDL syntax trees" to "PDDL syntax trees similar to d*".

Why cosine similarity on action descriptions rather than full domain text? Full domain PDDL text contains low-level syntax (parentheses, keywords) that dominates the embedding but carries little semantic signal. Abstract action descriptions ("UAV moves from one location to another") capture the functional structure of the domain -- what the actions do, not how they are written. This yields more structurally relevant retrievals.

7.3 Decomposition -- Why Action-by-Action Beats Monolithic Generation

Principle 3 -- Error Probability Under Decomposition $$ P_{\text{error}}^{\text{monolithic}} \approx 1 - (1 - p_{\text{action}})^{|\mathcal{A}|} \approx |\mathcal{A}| \cdot p_{\text{action}} \quad \text{(for small p)} $$ $$ P_{\text{error}}^{\text{decomposed}} \approx \sum_{k=1}^{|\mathcal{A}|} (1 - r)^{r_k} \cdot p_{\text{action}} \quad \text{(with retry mechanism)} $$
$p_{\text{action}}$ = baseline per-action error probability  |  $|\mathcal{A}|$ = number of actions  |  $r$ = probability that a retry corrects the error (syntax feedback effectiveness)  |  $r_k$ = number of retries for action k  |  Key insight: Monolithic error scales linearly with domain size. Decomposed generation adds an exponential damping factor $(1-r)^{r_k}$ per action, making total error sub-linear in domain size.
System-Level Synergy
The three components form a mutually reinforcing system: RAG provides structural templates that reduce the LLM's generation space; CoT decomposes each action into manageable reasoning steps; syntax feedback catches and corrects residual errors. Remove any one and performance degrades: without RAG, CoT alone still produces semantically incorrect actions; without CoT, syntactic error rate increases (Format vs FCoT); without syntax feedback, even correct reasoning can produce malformed PDDL.

9. Potential Flaw & Extensions

9.1 Limitations

  • Assumptions: (1) The curated UAV PDDL dataset covers the domain generation space -- if a user's task is fundamentally unlike any dataset domain, retrieval quality degrades. The dataset (30 domains) covers common UAV patterns but may not generalize to entirely novel task structures. (2) SPAR assumes the natural language description is complete and unambiguous -- if the user omits critical constraints (e.g., battery limits, no-fly zones), SPAR has no mechanism to detect these omissions.
  • Applicability scope: SPAR is designed for classical planning domains (deterministic, fully observable). It does not handle probabilistic effects, partial observability, or durative actions. Extending to temporal/numeric planning (PDDL 2.1) or probabilistic planning (PPDDL) would require significant changes.
  • Scalability: The embedding-based retrieval (all-mpnet-base-v2) works well for 30 domains but may not scale to thousands without approximate nearest neighbor search (e.g., FAISS). The LLM re-ranking step requires O(K) LLM calls per action.

9.2 Extension Directions

  • Dimension extension: From classical PDDL to PDDL 2.1 with durative actions and numeric fluents -- this would enable generation of domains with time-dependent constraints (e.g., "UAV must complete survey within 30 minutes").
  • Condition relaxation: (1) Develop a problem instance generator that, given a generated domain, automatically creates diverse problem instances. (2) Use the generated domain to bootstrap an active learning loop -- execute plans, detect semantic errors, and feed corrections back to the LLM.
  • Constraint addition: (1) Integrate formal verification of generated domains against natural language specifications (e.g., checking that "UAVs never run out of battery" is implied by the domain's preconditions/effects). (2) Add multi-modal input -- generate domains from visual demonstrations using vision-language models.

10. Motivation -- First-Principles Derivation

10.1 Idea Derivation Chain

  1. What is the essential problem?
    UAV autonomy requires planning -- but planning requires a domain model (PDDL). Writing PDDL domains is a specialized skill that most UAV operators lack. The bottleneck is not planning algorithms (Fast Downward works well) but domain authoring -- translating operational knowledge into formal PDDL syntax.
  2. How have others addressed this? Why is it insufficient?
    Three approaches: (i) manual authoring by PDDL experts -- does not scale; (ii) template-based generation -- inflexible, cannot handle novel task structures; (iii) learning from demonstrations -- requires extensive data collection per domain. None enables a non-expert to describe a UAV mission in natural language and receive a working PDDL domain.
  3. What if LLMs could write PDDL?
    LLMs are trained on code and structured text -- they should be capable of generating PDDL. But early experiments show high error rates: LLMs produce syntactically invalid PDDL, hallucinate predicates, and confuse action semantics. The core issue is that one-shot generation exceeds the LLM's single-pass reasoning capacity.
  4. What makes this task uniquely difficult for LLMs?
    PDDL requires: (a) formal type consistency across the entire domain; (b) correct predicate signatures used consistently across actions; (c) logically sound preconditions and effects. These are global constraints -- an error in action-1's predicate definition breaks action-5's usage. LLMs generate left-to-right and cannot enforce global consistency in one pass.
  5. What is the simplest solution that could work?
    Decompose: (1) provide a relevant example to anchor the structure (RAG); (2) generate actions one at a time with explicit reasoning (CoT); (3) validate each action before moving to the next (syntax feedback). This mirrors how a human writes PDDL -- look at similar domains, think carefully about each action, and test/debug incrementally.

10.2 One-Sentence Motivation

One-Sentence Motivation
PDDL domain authoring is the bottleneck that prevents UAV operators from using automated planning -- SPAR removes this bottleneck by letting users describe their mission in natural language and having an LLM generate the PDDL domain, using retrieval of similar examples as structural guidance, step-by-step reasoning for each action, and automated syntax correction for reliability.

10.3 How You Would Arrive at This Idea

  • Step 1 -- Identify the bottleneck: In the UAV autonomy pipeline (perception, planning, control), planning is algorithmically mature but domain modeling is the human-intensive bottleneck. Every new UAV mission requires writing a new PDDL domain.
  • Step 2 -- Recognize LLM capability and limitation: LLMs can generate structured text (code, JSON, XML). PDDL is structured text. But initial attempts show that LLMs struggle with PDDL's strict syntax and global consistency constraints. The capability exists but needs to be harnessed correctly.
  • Step 3 -- Design the decomposition: Instead of "generate complete domain," break it into "retrieve example, generate types/predicates, generate action-1, validate, generate action-2, validate, ..." Each sub-task is smaller and has clearer success criteria.
  • Step 4 -- Add retrieval for structural guidance: The decomposition still leaves semantic ambiguity (what predicates should this domain have?). Providing a similar, correct example resolves this by showing the LLM a valid structural template.
  • Step 5 -- Build the dataset and evaluation framework: To systematically evaluate, you need a diverse dataset with ground-truth domains. The 30-domain UAV PDDL dataset serves both as the retrieval corpus and the evaluation benchmark.

11. Key Takeaways

11.1 Lessons Learned

  • Methodology level: This is a decomposition-first approach to LLM-based structured generation. The key insight -- that monolithic generation fails for tasks with global consistency constraints, and that decomposition + retrieval + iterative validation solves this -- generalizes beyond PDDL. Any task requiring LLMs to produce globally consistent structured output (hardware description languages, robot configuration files, mission scripts) can benefit from this template.
  • Technical level: (1) Action descriptions as retrieval keys -- embedding the functional description of what actions do, rather than raw PDDL text, yields better retrieval quality because it captures structural semantics rather than surface syntax. (2) LLM-friendly error feedback -- translating parser errors into natural language explanations enables the LLM to self-correct, closing the loop between generation and validation. (3) Dynamic fluent lists -- explicitly tracking defined predicates/types and injecting them into subsequent prompts solves the cross-action consistency problem.
  • Experiment/writing level: The four-metric evaluation (syntax errors, executability, feasibility, interpretability) provides a comprehensive view of generation quality. Most papers stop at syntax or executability; adding feasibility and interpretability captures semantic correctness. The case studies (UAV Watering, MovableObs) ground the quantitative results in concrete UAV scenarios.

11.2 Directly Applicable to My Research

  • For UAV task planning: SPAR's generated PDDL domains can directly enable automated planning for a wide range of UAV missions. The 30-domain dataset covers common UAV patterns -- most new missions will have a structurally similar domain in the dataset, ensuring good retrieval quality.
  • For multi-robot coordination: The MovableObs case study (UAV-UGV collaboration) demonstrates SPAR's ability to handle multi-agent domains with interdependent actions. This can be extended to multi-UAV coordination scenarios (formation flight, cooperative surveillance, task allocation).
  • For human-robot interaction: SPAR's natural language interface democratizes UAV planning -- operators without PDDL expertise can describe missions in plain language and receive executable plans. This is a step toward conversational robot tasking, where the planning domain adapts dynamically as the user refines their description through dialogue.
